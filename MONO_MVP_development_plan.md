# MONO 애플리케이션 고도화 개발 계획서

## 1. 개발 목적

MONO MVP를 단순 시연용 서비스에서 마케팅 분석이 가능한 운영형 MVP로 고도화한다.

이번 개발의 핵심 목적은 다음과 같다.

1. 기술자, 현장 리더, 기업 사용자를 구분하여 가입·프로필·공고·팀 데이터를 수집한다.
2. 기업이 실제 채용 공고와 현장 작업 요청을 등록할 수 있게 한다.
3. 현장 리더가 팀과 가동 가능한 인력을 등록할 수 있게 한다.
4. 주요 사용자 행동 이벤트를 수집하여 마케팅 분석 웹에서 확인한다.
5. 관리자 앱에서 사용자, 기업, 공고, 이벤트, 관심 기능을 관리한다.
6. 아직 구현하지 않는 기능은 안내 팝업과 관심 신청으로 처리해 수요를 검증한다.

## 2. 서비스 구성

| 서비스 | URL | 목적 |
|---|---|---|
| 사용자 웹/앱 | `mono.dojiung.com` | 기술자·현장 리더 가입, 프로필 작성, 공고 조회, 팀 등록 |
| 마케팅 분석 웹 | `mono.dojiung.com/analytics` | 전환율, 재방문율, 공고 행동, 관심 기능 분석 |
| 관리자 앱 | `mono.dojiung.com/amono` | 사용자, 기업, 공고, 이벤트, 신고, 관심 신청 관리 |
| 기업용 웹 | `mono.dojiung.com/partner` | 기업 등록, 공고 등록, 현장 작업 요청, 후보 검토 |

※ `mono.djiung.com/partner`는 오타 가능성이 있으므로 실제 배포는 `mono.dojiung.com/partner` 기준으로 개발한다.

## 4. 개발 원칙

### 4-1. UI·UX 원칙

1. 운영 단계 수준의 완성도를 목표로 한다.
2. 모바일 우선으로 개발하되 PC 웹에서도 정상 작동해야 한다.
3. 기술자 입력은 선택형, 카드형, 단계형으로 구성한다.
4. 기업용 화면은 복잡한 SaaS보다 공고 등록과 후보 검토를 빠르게 할 수 있게 한다.
5. 구현 전 기능은 비활성 버튼이 아니라 안내 팝업과 관심 신청으로 연결한다.
6. 모든 주요 CTA는 이벤트 로그가 남아야 한다.

### 4-2. 데이터 원칙

1. 개인정보, 프로필 데이터, 행동 이벤트 데이터를 구분한다.
2. 사용자 유형별 권한을 분리한다.
3. 모든 핵심 행동은 이벤트 로그로 저장한다.
4. 관리자와 마케팅 분석 웹에서 이벤트 수집 상태를 확인할 수 있어야 한다.
5. 향후 Trust Data Lake 확장을 고려해 테이블 구조를 설계한다.

## 13. 최종 개발 방향

MONO 애플리케이션은 기술자와 기업을 단순 연결하는 앱이 아니라, 현장 기술자의 경험과 기업의 현장 운영 데이터를 축적하는 운영형 MVP로 고도화한다.

초기에는 가입, 프로필, 공고, 기업 요청, 팀 등록, 이벤트 분석을 중심으로 개발한다. 이후 집결, 이동, 출역, 정산, 팀 커뮤니티, 평가, 재요청 데이터를 축적하여 MONO의 핵심 차별성인 산업 신뢰 데이터 인프라를 구축한다.
